2) BACKEND (EXPRESS) — ONLY ONBOARDING ROUTES + ACCESS GATE
========================================================
Add routes file: `server/src/routes/onboarding.ts` and mount at `/onboarding`.

A) Worker Application
- POST /onboarding/worker-application   (worker only)
  - Validate required fields (A–R required items)
  - Create WorkerApplication (or upsert if already exists and status=submitted)
  - Set user.onboardingStatus = APPLICATION_SUBMITTED
  - Store submitIp and submitUserAgent from request headers if available
  - Return { applicationId, status, onboardingStatus }

- GET /onboarding/worker-application/me (worker only)
  - Return worker’s application + status + onboardingStatus

B) HR/Admin Review (minimal admin endpoints ONLY for these forms)
Create routes file: `server/src/routes/adminOnboarding.ts` and mount at `/admin/onboarding`.

- GET /admin/onboarding/applications (hr/admin)
  - List applications with filters: status, date range, search by name/email; paginate

- GET /admin/onboarding/applications/:id (hr/admin)
  - Application detail

- PUT /admin/onboarding/applications/:id/approve (hr/admin)
  - Set application.status=approved, reviewedById, reviewedAt
  - Set user.onboardingStatus=AGREEMENT_PENDING
  - Return updated records

- PUT /admin/onboarding/applications/:id/reject (hr/admin)
  - Set application.status=rejected, reviewedById, reviewedAt
  - Set user.onboardingStatus=APPLICATION_REJECTED

C) Agreement Template + Acceptance
- GET /onboarding/agreement/template/active (worker only; only if onboardingStatus is AGREEMENT_PENDING or higher)
  - Return active template (id, version, title, bodyText, lastUpdated, effectiveDate)

- POST /onboarding/agreement/accept (worker only; only if onboardingStatus=AGREEMENT_PENDING)
  Body:
    - templateId
    - acceptedFullName
    - initials: {s19_1,s19_2,s19_3,s19_4,s19_5}
    - dateLocal (YYYY-MM-DD)
    - timeZone
    - iAgree (boolean)
  Behavior:
    - Require iAgree true, acceptedFullName non-empty, all initials non-empty
    - Create acceptance with templateBodySnapshot = template.bodyText and templateVersion
    - Create AgreementSubmission (submittedToAdminAt=now)
    - Set user.onboardingStatus = AGREEMENT_ACCEPTED
    - Return onboardingStatus

- GET /onboarding/agreement/me (worker only)
  - Return latest acceptance + submission status (if any)

D) Admin Agreement Review
Add minimal routes in `server/src/routes/adminAgreements.ts` mounted at `/admin/agreements` (admin only):
- GET /admin/agreements/submissions
  - list submissions (submitted/reviewed) + paginate
- GET /admin/agreements/submissions/:id
  - detail includes acceptance + template snapshot
- PUT /admin/agreements/submissions/:id/review
  - mark reviewed; adminReviewedAt/by; status=reviewed
  - optionally set worker onboardingStatus=ONBOARDED

E) ACCESS GATE (critical)
Create middleware `server/src/middleware/requireWorkerOnboardingComplete.ts`:
- If req.user.role !== "worker" => next()
- If req.path starts with "/onboarding" or "/auth" => next()
- Allow read-only "/profile/me" if it exists
- If user.onboardingStatus IN ("AGREEMENT_ACCEPTED","ONBOARDED") => next()
- Else respond 403:
  { code:"WORKER_ONBOARDING_INCOMPLETE", onboardingStatus:user.onboardingStatus }

Mount this middleware for all authenticated routes (or globally after auth middleware) so workers cannot access the rest of the app until agreement accepted.
