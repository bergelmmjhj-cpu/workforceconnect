5) CORE MODULES (MVP BEHAVIOR)
========================================
A) Worker Requests (Client -> HR workflow)
- Client creates WorkerRequest:
  - roleNeeded (string or enum)
  - shiftStartTime, shiftEndTime (UTC stored)
  - locationMajorIntersection (string; do not store precise addresses)
  - payStructure (string, with visibility rules)
  - notes (text)
  - status: draft, submitted, reviewing, assigned, in_progress, completed, cancelled
  - hrAssignedId optional
  - slaDeadline (UTC)
- HR can:
  - list submitted requests
  - set hrAssignedId
  - update status and create shifts from requests
  - SLA tracking: show upcoming deadlines and breaches in dashboard “Smart To-Do”

B) Shift Management
- HR creates Shift linked to WorkerRequest
- HR assigns workers (via ShiftAssignment)
- Worker sees assigned shifts
- Client sees shifts associated with their requests

C) TITO
- Worker submits:
  - timeIn (server-validated; ignore client-provided time if it’s wildly off; store UTC)
  - timeOut
  - timeInLocation/timeOutLocation (optional coarse string)
  - verificationMethod (enum: "gps", "selfie_placeholder", "manual", "other")
- HR/Client can approve/dispute:
  - approvedBy, approvedAt, disputedBy, disputedAt
- Audit trail:
  - store audit entries (JSON) OR separate AuditLog rows

D) Messaging (Quo-like)
- Conversations between:
  - Client <-> HR
  - HR <-> Worker
  - Admin oversight optional (admin can access if required)
- Realtime Socket.io:
  - New message event broadcasts to conversation participants
  - Notification badge updates in realtime
- Consistency requirement (avoid “different users see different data”):
  - Use consistent DB reads
  - Do not rely on client caching only
  - When message created: write DB first, then emit socket event, then return API response
  - On client: refetch on socket events or update local state reliably

E) Notifications
- Store notifications in DB:
  - title, message, type, isRead, actionUrl
- Trigger on:
  - request created/submitted
  - worker assigned
  - shift starting soon (basic SLA job; can be a simple server interval in MVP)
  - TITO submitted
  - approval needed
  - message received
- Email stubs: console.log(...) only

F) Reporting + CSV Export
- /reports GET returns:
  - shifts per worker
  - hours worked (based on approved TITO where possible)
  - approvals pending
  - SLA breaches
- /exports/shifts-per-worker.csv returns CSV
