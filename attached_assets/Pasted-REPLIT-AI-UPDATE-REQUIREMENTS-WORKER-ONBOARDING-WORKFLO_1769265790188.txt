A) UPDATED WORKER ONBOARDING WORKFLOW (MUST IMPLEMENT)
========================================
Goal:
Before a worker can access the main application features, they must:
1) Submit a Worker Application Form
2) Get Approved by HR or Admin
3) Accept/Sign the Subcontractor Agreement
4) The signed agreement is submitted to Admin for review/record
Only after step 3 is completed (and optionally step 4 marked “received/reviewed”) can the worker access normal worker features.

Define “Access”:
- Worker cannot access: dashboard, shifts, TITO, messaging, etc. until onboarding passes.
- Worker CAN access: the onboarding portal pages (application + agreement) and view onboarding status.

Implementation:
- Add an “onboardingStatus” state machine for worker users:
  - NOT_APPLIED
  - APPLICATION_SUBMITTED
  - APPLICATION_APPROVED
  - APPLICATION_REJECTED
  - AGREEMENT_PENDING
  - AGREEMENT_ACCEPTED
  - ONBOARDED (optional final state; can be set after admin acknowledges receipt)
Rules:
- When a worker registers, their onboardingStatus = NOT_APPLIED.
- Worker completes application form => onboardingStatus becomes APPLICATION_SUBMITTED.
- HR/Admin approves application => onboardingStatus becomes APPLICATION_APPROVED and agreement becomes available; set onboardingStatus = AGREEMENT_PENDING.
- Worker must accept agreement (e-sign checkbox + typed full name + timestamp) => onboardingStatus becomes AGREEMENT_ACCEPTED.
- Once agreement is accepted, automatically create an “AgreementSubmission” record sent to Admin inbox; Admin can mark it “RECEIVED/REVIEWED”.
- Worker is allowed into main app when onboardingStatus IN (AGREEMENT_ACCEPTED, ONBOARDED) depending on your design. Prefer: allow access after AGREEMENT_ACCEPTED, and keep ONBOARDED for admin tracking.

Access Control:
- Backend middleware must block all worker routes except:
  - /onboarding/*
  - /auth/*
  - /profile/me (read-only)
- Frontend route guard must redirect workers who are not onboarded to onboarding pages.

C) API UPDATES (EXPRESS + RBAC) — MUST IMPLEMENT
========================================
Onboarding Endpoints:
- POST /onboarding/worker-application
  Worker submits application form (role=worker only).
  Sets worker onboardingStatus => APPLICATION_SUBMITTED.
  Creates WorkerApplication(status=submitted).
  Notifies HR/Admin.

- GET /onboarding/worker-application/me
  Worker fetches their application + status.

- PUT /onboarding/worker-applications/:id/approve
  HR/Admin approves; sets WorkerApplication.status=approved; sets worker onboardingStatus => AGREEMENT_PENDING.
  Notify worker.

- PUT /onboarding/worker-applications/:id/reject
  HR/Admin rejects; sets WorkerApplication.status=rejected; sets worker onboardingStatus => APPLICATION_REJECTED.
  Notify worker.

- GET /onboarding/agreement/template/active
  Worker fetches active agreement template (only if onboardingStatus is AGREEMENT_PENDING or higher).

- POST /onboarding/agreement/accept
  Worker accepts agreement:
  Body: { templateId, acceptedFullName }
  Server sets acceptedAt=now UTC, stores metadata (ip/user-agent if available).
  Sets worker onboardingStatus => AGREEMENT_ACCEPTED (or ONBOARDED if you choose).
  Creates AgreementSubmission(status=submitted, submittedToAdminAt=now).
  Notify Admin.

- GET /onboarding/agreement/me
  Worker sees acceptance record + submission status.

Admin Review:
- GET /admin/agreements/submissions
  Admin lists submissions (status=submitted/reviewed).

- PUT /admin/agreements/submissions/:id/review
  Admin marks reviewed; sets status=reviewed, adminReviewedAt, adminReviewedById.
  Optionally set worker onboardingStatus => ONBOARDED.

Existing Endpoints:
Keep all previous endpoints (/auth, /users, /requests, /shifts, /tito, messaging, notifications, reports, exports) but enforce onboarding gate for workers.

Middleware:
- auth middleware: verifies JWT
- rbac middleware: role checks
- onboarding middleware:
  If user.role=worker and onboardingStatus not in allowed states, block access to non-onboarding endpoints with 403 and a clear error code:
    { code: "WORKER_ONBOARDING_INCOMPLETE", onboardingStatus: "..." }
